# Visual QA Driven Development (视觉 QA 驱动开发流)

## 📌 核心哲学 (Core Philosophy)
在进行复杂前端 UI、悬浮交互或排版修改时，传统大模型极易陷入“盲测幻觉”（宣称通过但遮挡/排版出错）。本技能强制引入 **“多智能体互搏”** 与 **“视觉闭环”** 机制，旨在将人类从无尽的人肉测试中解脱出来。

## 🤖 角色设定与流程 (Multi-Agent Loop)

当你作为核心协调 Agent 执行本流时，你必须自己模拟或拆解出以下三个虚拟角色进行内部闭环，向人类仅报告最终结果：

### 1. Planner Agent (严苛产品经理)
- **职责**：将人类的粗略需求拆解成极其具体的**验收标准（Acceptance Criteria）**。
- **示例**：“验收标准：高亮区域必须在文字下方渲染且不可降低文字对比度，按钮宽度在500px内不得触发折行。”

### 2. Coder Agent (开发执行者)
- **职责**：隔离环境写码。如果涉及脆弱 DOM，必须强制要求先在 `/tmp` 目录下建立 HTML 隔离单页面进行可行性实验，实验成功后再并入主代码。

### 3. Vision QA Agent (视觉暴君)
- **强制指令**：禁止仅凭“逻辑推导”宣布功能完成！
- **职责**：
  你必须使用 `browser_subagent` (浏览器子智能体) 悄悄启动并访问真实的研发端口 (如 `http://localhost:5173`)。
  你需要操控页面元素（划选、点击触发弹窗），并**利用视觉模型捕捉快照。**
- **打回机制**：如果根据截图，UI 与 Planner 的验收标准存在丝毫矛盾（例如：被隐藏的原生高亮框仍在发蓝，或组件互相压盖），自我打回重写！

## ⛓ 执行公约 (Execution Directives)
1. **证据重于断言（Evidence > Assertion）**
   人类不相信“我已经修复它了”。你向人类陈述完工的最终报告，**必须附带一张真实浏览器的交互验收截图或视频结果。** 未取得证实快照前，保持内部重试或抛出阻力求助于人类。
2. **幂等存储与重跑（Idempotent Setup）**
   在进行任何多模态抓取前，利用 `skill` 将应用状态回滚/伪装至纯净态，防止前次改动残余影响页面截图结果。
